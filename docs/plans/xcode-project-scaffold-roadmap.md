Xcode Project Scaffold 完整後續開發計劃

修訂:2026-07-24

本版依 grilling session 的九項決策全面修訂:

1. CocoaPods 由「一級方案」降級為「supported, not first-class」,聚焦企業情境
2. 設定檔名維持 scaffold.yml(前版的 xscaffold.yaml 為筆誤)
3. 不引進 create,互動式入口維持 new
4. init 退役,由 generate 接手非互動生成
5. --force 收窄為兩級規則,不移除
6. v0.4 維持 Distribution + Preview Workflow 同版
7. 資料模型收斂為五型別,取消 Raw/Resolved 拆分
8. preset 一詞讓給規模等級,平台組合改稱 variant
9. 詞彙翻轉於 v0.4 一次到位(--variant 上線,--preset 於 v0.7 回歸)

排程原則:版本順序可調整,對外推廣的時點在接近全部完成之後。

⸻

1. 專案定位

xcode-project-scaffold 的目標,是成為 Xcode 專案領域的 create-next-app。

使用者應能透過簡單指令,快速建立結構一致、可以編譯、可以測試,並具備基本開發工具與依賴管理能力的 iOS 或 macOS 專案。

理想使用方式:

brew install g761007/tap/xscaffold
xscaffold new MyApp

工具應協助使用者完成:

1. 選擇平台與 UI 技術
2. 選擇專案架構
3. 選擇測試框架
4. 選擇 SPM、CocoaPods 或混合依賴
5. 建立 scaffold.yml
6. 預覽解析後設定
7. 預覽預計建立的檔案與執行指令
8. 讓使用者確認、修改、儲存草稿或取消
9. 確認後才正式生成專案
10. 執行 XcodeGen
11. 安裝依賴
12. 初始化 Git
13. 視設定執行 Build Validation
14. 開啟 .xcodeproj 或 .xcworkspace

專案下一階段的核心方向為:

Preview First, Generate After Confirmation.
SPM First, CocoaPods Supported.
Distribution First, More Architectures Later.

3. 核心產品原則

3.1 只建立專案,不持續管理專案

xscaffold 的主要責任是建立新專案。

生成完成後:

* project.yml 是 XcodeGen 專案設定來源
* .xcodeproj 或 .xcworkspace 是主要開發入口
* 開發者自行修改與維護專案
* xscaffold 不追蹤生成檔案後續變更
* xscaffold 不自動將新模板同步到既有專案

暫不提供:

xscaffold add
xscaffold migrate
xscaffold regenerate
xscaffold update-project
xscaffold upgrade-project

避免引入:

* 檔案 Ownership
* Existing File Merge
* Template Migration
* Template Version Synchronization
* Checksum Tracking
* Conflict Resolution

⸻

3.3 scaffold.yml 是 Scaffold Manifest

scaffold.yml 應保留在生成後的專案內,但它不是持續管理專案的唯一真實來源。

它代表:

專案建立當下所使用的 Scaffold Manifest。

用途包括:

* 記錄專案建立方式
* 供團隊進行 Code Review
* 供 AI Agent 理解初始專案設定
* 重現模板與生成問題
* 提交 Bug Report
* 在空目錄中重建相同的起始專案
* 保存依賴管理、測試與環境選擇

建議檔頭:

# Generated by xscaffold.
#
# This file records the configuration used to create the project.
# xscaffold does not manage or migrate the project after generation.
schemaVersion: 1

Schema Version 規則:

* 檔案缺少 schemaVersion 欄位時,一律視為 version 1,不報錯
* 不支援的版本回報 SCHEMA_VERSION_UNSUPPORTED

若開發者後續手動修改 project.yml 或 Xcode 專案,scaffold.yml 不會自動同步。

3.4 SPM 優先,CocoaPods 供需要的團隊使用

生態系事實(本節為排程依據,必須隨事實更新):

* CocoaPods 自 2024 年 8 月起進入 maintenance mode
* 公開 trunk 將於 2026 年 12 月 2 日永久轉為 read-only
* 之後公開 trunk 不再接受新 pod 或版本更新
* 既有 pod 仍可安裝;Private Specs Repo 不受影響

因此,對一個「建立全新專案」的工具而言,CocoaPods 的合理定位是:

SPM is the default. CocoaPods is supported for teams that need it —
existing enterprise stacks, private pods, and pod-only third-party SDKs.

依賴管理策略:

* Swift Package Manager(預設)
* CocoaPods
* SPM 與 CocoaPods 混合模式
* 無依賴管理工具

CocoaPods 的主要適用情境:

* 公司統一使用 CocoaPods 的技術棧
* Private Pod 與公司內部元件庫
* 多個 Specs Repository
* 第三方 SDK 只提供 Pod
* SPM 與 CocoaPods 混合依賴
* 未來的 Objective-C 或混合語言專案

支援原則:

* Validator 不得人為限制語言或 UI 技術,例如「Swift projects cannot use CocoaPods」
* 但文件與預設選項一律推薦 SPM
* CocoaPods 進階功能(Target-specific Pods、Structured Post-install、CI Cache)降為按需求實作,不做預先建設

Validator 應檢查:

* Pod 是否支援目標平台
* Pod Deployment Target
* Target 是否存在
* Pod Source 是否有效
* Static/Dynamic Linkage 設定
* 重複依賴
* 與 SPM 的依賴衝突
* Workspace 是否成功產生
* Workspace Build 是否成功

4.2 xscaffold new

用途:

一次完成互動設定、預覽、確認與生成。互動式入口只有這一個。

使用方式:

xscaffold new MyApp

內部流程:

互動式問答
→ DraftConfiguration
→ Resolve Defaults
→ Validate
→ Build Generation Plan
→ Preview
→ Confirm
→ Write scaffold.yml
→ Generate
→ Verify

支援:

xscaffold new MyApp --advanced
xscaffold new MyApp --yes
xscaffold new MyApp --variant ios-uikit
xscaffold new MyApp --dependency-manager cocoapods
xscaffold new MyApp --open
xscaffold new MyApp --validate-build
xscaffold new MyApp --skip-git
xscaffold new MyApp --force

其中:

* --variant 是非互動選擇平台組合的捷徑(見 §17.1);對應的問答直接跳過
* --yes 跳過人工確認
* --yes 不跳過 Validation
* --yes 不跳過 Plan Resolution
* --yes 不繞過 Output Directory 規則(見 §13.3)
* --validate-build 沿用既有 flag 名稱,生成後執行 Build Validation
* 非互動模式不得出現 Prompt
* v0.7 之後另支援 --preset minimal|standard|production(規模,見 §17.2)

一行生成(取代舊 init --preset 的用法):

xscaffold new MyApp --variant ios-uikit --yes

⸻

4.4 xscaffold init 的退役

今天的 init 直接生成專案(--preset 或 --config)。它的兩個職責由新指令接手:

init --config existing.yml   →  generate --config existing.yml
init MyApp --preset ios-uikit →  new MyApp --variant ios-uikit --yes

互動式的 manifest-only 需求(先寫設定、不生成)由 new 的
「Save scaffold.yml and Exit」涵蓋;非互動的 manifest 需求由
config example 涵蓋。因此不設立獨立的 manifest-only 指令。

退役步驟見 §5。

7. 草稿與資料模型

Interactive Answers

→ ValidatedConfiguration
→ GenerationPlan

7.1 狀態模型

enum ScaffoldPhase {
    case draft
    case normalized
    case validated
    case planned
    case confirmed
    case generating
    case generated
    case failed
    case cancelled
}

ScaffoldPhase 是進度回報與錯誤契約(§23)的輸出詞彙,
不是每個 case 都需要對應一個型別。

⸻

7.2 資料模型

五個型別,不多不少:

struct DraftConfiguration        // 新增
struct ProjectConfiguration      // 既有
struct ValidatedConfiguration    // 新增(薄 wrapper)
struct GenerationPlan            // 既有
struct GenerationResult

DraftConfiguration

* 保存互動中尚未完成的資料
* 欄位全部 optional,可以不完整
* 不可直接傳入 Generator

ProjectConfiguration

* 既有型別,對應 scaffold.yml 結構
* 預設值解析照舊集中在 ConfigurationDefaults
* 「使用者寫了什麼」由 scaffold.yml 本身記錄;
  「解析後長什麼樣」由 plan --resolved-config 直接渲染本型別。
  不另設 Raw/Resolved 兩個近乎相同的型別與對映層。

ValidatedConfiguration

* 包住通過驗證的 ProjectConfiguration 的薄 wrapper
* init 只開放給 ConfigurationValidator
* Generator 只能接收此型別 — 進入生成的路徑越多
  (new 互動、generate、edit 迴圈),這個編譯期保證越值錢

GenerationPlan

記錄:

* Files
* Directories
* Targets
* Dependencies
* External Commands
* Output Container
* Warnings
* Output Path
* Template Sources
* 將被覆蓋的既有檔案(--force 情境)

GenerationResult

記錄:

* 實際產生的檔案
* 外部命令結果
* .xcodeproj 路徑
* .xcworkspace 路徑
* Build Validation
* Rollback State
* Duration
* Warning

核心規則:

* Draft 不可傳入 Generator
* Generator 只接受 ValidatedConfiguration

9. Dependency Management

9.1 模式

dependencyManagement:
  mode: none
dependencyManagement:
  mode: spm
dependencyManagement:
  mode: cocoapods
dependencyManagement:
  mode: mixed

規則:

* none 不載入 SPM 或 CocoaPods 設定
* spm 只使用 SPM
* cocoapods 只使用 CocoaPods
* mixed 同時使用 SPM 與 CocoaPods
* mixed 必須執行重複依賴檢查
* 同一套 Library 不應同時由 SPM 與 CocoaPods 引入

10.2 Target Products

  targets:
    app:
      products:
        - package: Alamofire

11. CocoaPods 支援

11.1 支援範圍

CocoaPods 支援:

* iOS SwiftUI Swift 專案
* iOS UIKit Swift 專案
* macOS SwiftUI Swift 專案
* macOS AppKit Swift 專案
* 未來的 Objective-C 專案
* 未來的 Swift/Objective-C 混合專案

Validator 不做語言或 UI 技術的人為限制;
但產品層級的推薦與預設永遠是 SPM(§3.4)。

功能分級:

* 基本支援(v0.5):mode: cocoapods、Podfile 生成、pod install、
  Workspace 辨識、Doctor 檢查
* 企業情境(v0.6):Bundler、Version Pinning、多 Sources、Private Pods
* 按需求(P3,不預先建設):Target-specific Pods、
  Structured Post-install、CocoaPods CI Cache

11.2 基本設定

cocoapods:
  runInstall: true
  useBundler: true
  version: "1.16.2"
  useFrameworks: false
  linkage: static
  generateWorkspace: true

欄位:

欄位	說明
runInstall	是否在生成後執行 pod install
useBundler	是否透過 Bundler 執行
version	CocoaPods 版本
useFrameworks	是否產生 use_frameworks!
linkage	static 或 dynamic
generateWorkspace	是否預期產生 Workspace

當 useFrameworks 為 false 時,linkage 應被忽略或產生 Warning。

enum ProjectContainer {

13.2 失敗處理

任何步驟失敗時:

* 不留下半完成的正式專案
* 清理 Temporary Directory
* 回報失敗 Phase
* 回報 External Command
* 回報 Exit Code
* 提供 Recovery Suggestion
* JSON Output 必須保持合法
* 不可吞掉 CocoaPods、XcodeGen 或 Xcodebuild stderr

⸻

13.3 Output Directory 兩級規則

一律允許(不需要任何 flag):

* 不存在的目錄
* 空目錄
* 只包含 scaffold.yml 的 Manifest 目錄

硬拒絕(任何 flag 都不能繞過):

* 已包含 .xcodeproj
* 已包含 .xcworkspace
* 已包含 project.yml
* 已包含來源碼的目錄

錯誤:

Generation refused because the output directory contains
existing project files.
xscaffold creates new projects and does not update existing projects.

軟拒絕(--force 可繞過):

* 非空、但沒有上述專案標記的目錄
* 典型情境:GitHub 起手式 — gh repo create 勾選
  README + LICENSE + .gitignore 後 clone,在裡面 scaffold

--force 的語意:

* 允許進駐無專案標記的非空目錄
* 撞名檔案會被覆蓋,plan 與 preview 必須明確列出將被覆蓋的檔案
* 不清空目錄
* 永遠不能把「硬拒絕」降級

⸻

13.4 Template Conflict

生成前建立完整 File Manifest。

若兩個模板產生相同路徑:

Template conflict detected.
File:
  Sources/App/AppDelegate.swift
Provided by:
  Interfaces/UIKit
  Features/CustomLifecycle

禁止後載入模板靜默覆蓋前一個模板。

17. Variant 與 Preset

兩個正交的維度,各自有專屬的詞與 flag,不共用:

* Variant = 平台 × UI 技術(CLI:--variant,v0.4 起)
* Preset = 專案規模與預設功能(CLI:--preset,v0.7 起)

17.1 Variant

Variant 表示平台與 UI 技術,即現有四個名字:

ios-swiftui
ios-uikit
macos-swiftui
macos-appkit

* CLI 捷徑:new MyApp --variant ios-uikit
* scaffold.yml 內仍是 platform + interface 兩欄位,variant 只是 CLI 捷徑
* 這四個名字在 v0.4 前掛在 --preset 下;v0.4 起改掛 --variant,
  --preset 同時消失(§5)

17.2 Preset

Preset 表示專案規模與預設功能:

minimal
standard
production

於 v0.7 上線,屆時 --preset 以此語意回歸。

⸻

17.3 Minimal

* Minimal Architecture
* Dependency Management:None
* Unit Test 可選
* 無 Environment
* 無 CI
* 無 Lint
* 無 Format

適合:

* Prototype
* Sample
* 學習專案

31. 最終方向

xcode-project-scaffold 接下來不應以增加更多 Architecture Template 為主要目標。

真正需要優先完成的是:

1. 一行安裝
2. 一條指令建立
3. 先生成設定草稿
4. 正式生成前完整預覽
5. 允許修改、儲存或取消
6. 原子式安全生成
7. SPM 完整支援
8. CocoaPods 支援涵蓋需要它的團隊(企業、Private Pods、Pod-only SDK)
9. 自動辨識 Project 或 Workspace
10. 人類、CI 與 AI Agent 都能可靠使用

最終產品定位:

A preview-first, agent-friendly Xcode project scaffolder —
SPM-first, with CocoaPods support for teams that need it.
